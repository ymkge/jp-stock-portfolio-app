# 時系列データ管理基盤の強化：基本設計書

## 1. 背景と目的
現在、国内株式のテクニカル分析（75日移動平均、125日フィボナッチ等）に必要な遡及期間が不足しており、「安値圏（🌀）」シグナルが正しく機能していません。また、株式分割発生時のデータ整合性維持も課題となっています。
本計画は、サイトへの負荷を最小限に抑えつつ、永続的に時系列データを蓄積・活用し、正確な分析を可能にする基盤を構築することを目的とします。

## 2. 要件定義 (Requirements)

1.  **データの蓄積と保護**: 外部サイトの掲載制限期間（約1年）を超えて、DBに2年、3年と長期の株価推移を蓄積できること。
2.  **低負荷運用**: 毎回のAPI更新では最新1ページのみを取得。重い過去分取得は独立したメンテナンスツールに分離する。
3.  **株式分割への自動対応**: 分割・併合発生時、DB内の過去データを現在の尺度に自動で一括補正（スケーリング）すること。
4.  **計算精度の完全復旧**: 125日以上の有効な時系列データを確保し、全てのテクニカルシグナルを正常化する。

## 3. 基本設計 (Design)

### 3.1 データベース拡張 (Migration)
`daily_stock_history` テーブルを拡張し、JSONパースなしで高速な検索・補正を可能にします。
- **追加カラム**:
    - `close_price` (REAL): 終値（分割補正後の計算用）
    - `volume` (INTEGER): 出来高（分割判定の精度向上用）
- **移行処理 (Backfill)**: 既存の `data_json` から過去の株価を抽出し、新カラムに自動で埋め戻すスクリプトを実行。

### 3.2 メンテナンスツール (`sync_history.py`) [手動実行]
`portfolio.json` に登録された銘柄の履歴を管理する独立したツール。
- **増分・継ぎ足し取得**: DB内の最古/最新の日付を確認し、不足している過去期間のみを `page=N` を指定して段階的に取得。
- **分割検知と補正 (Scaling Logic)**: 
    1.  取得した過去データとDB内の同日データを照合。
    2.  乖離（例：5%以上）があれば分割発生と判定し、比率（係数）を算出。
    3.  DB内の当該銘柄の **全過去レコード** に対して係数を掛け合わせ、一括更新する。

### 3.3 メインアプリのロジック改善 (`scraper.py`)
- **ハイブリッド計算**: 「通信で取得した最新20日分」＋「DBに蓄積された過去N日分」をマージして移動平均等を算出。
- **安全性**: データが不足している場合は `is_reliable=False` とし、「🔘 判定不能」バッジを表示して誤認を防止。

## 4. リスクと対策 (Risk Management)

| リスク項目 | 内容 | 対策 |
| :--- | :--- | :--- |
| **株式分割の誤認** | 市場の暴落等を分割と勘違いする可能性。 | **過去の同じ日付** の株価同士を比較するため、当日の暴落に惑わされない。複数日照合で精度を高める。 |
| **データの「穴」** | 通信エラー等による欠損。 | DBの連続性をチェックし、欠損期間をピンポイントで埋めるロジックをメンテナンスツールに実装。 |
| **IP制限 (403)** | 大量取得による制限。 | メンテナンスツールでは1銘柄ごとに十分な待機時間を設ける。また、1ページ目照合で「変更なし」なら即終了する。 |
