# Yahoo!ファイナンス Next.js 刷新に伴うスクレイピング機能復旧計画

## 1. 背景と現状分析
Yahoo!ファイナンスの技術基盤が Next.js (App Router) へ移行したことにより、従来のデータ抽出元（`window.__PRELOADED_STATE__`）が廃止された。これにより、現在全銘柄で解析エラーが発生し、サーキットブレーカーが作動している。

### 判明した主な変更点
- **データ形式**: JSONから `self.__next_f.push` による断片的なストリーミング形式へ変更。
- **データ配置**: ページを跨いだデータの分離が加速。
    - **メインページ**: 株価・財務指標（PER/PBR等）のみ。
    - **時系列ページ**: 過去株価（移動平均用）がこちらに集約。
- **解析の難易度**: HTMLが巨大な1行（数十万文字）で構成されており、標準的なツールがフリーズしやすい。

## 2. 復旧フェーズと作業手順

### フェーズ1：データ構造の精密マッピング（調査）
**目標**: Next.jsの分割データから、必要な全項目を抽出するための「座標（正規表現ルール）」を確定させる。

1. **安全なデータサンプリング**:
    - フリーズを避けるため、Pythonのストリーム処理を用いて、特定のキーワード（`indicators`, `histories`, `priceBoard`）周辺の文字列のみを surgical read（外科的抽出）する。
2. **キー名対応表の作成**:
    - 旧JSON構造と新Next.js構造の1対1対応表を定義する。
    - 例: `totalPrice` (旧) → `indicators.totalPrice.value` (新)

### フェーズ2：次世代パースエンジンの開発（試作）
**目標**: 複雑なNext.js通信を「一つの仮想JSON」として復元する共通部品を完成させる。

1. **`NextDataParser` ユーティリティの実装**:
    - `self.__next_f.push` の全断片を回収し、インデックス順に結合するロジック。
    - 多重エスケープ文字（`\\\\\\"` 等）の正規化。
2. **プロトタイプによる多銘柄検証**:
    - 5銘柄以上の異なる資産（国内株、米国株、投信、指標）で全項目が取れるか、スタンドアロンのスクリプトでテスト。

### フェーズ3：`scraper.py` の全面リプレース（統合）
**目標**: 既存のスクレイパークラスを新エンジンへ繋ぎ込み、APIの互換性を保証する。

1. **マルチページ巡回の実装**:
    - 効率化のため、まず `/history` ページを取得し、不足分を `/quote` で補う「スマート・フェッチ」を実装。
2. **互換レイヤーの構築**:
    - `app.py` が期待する旧来の辞書形式を維持し、ロジック側の変更を不要にする。
3. **バリデーターの実装**:
    - 時系列データが計算に十分な件数（例: 75件）あるかチェック。

### フェーズ4：最終検証と本番移行（安定化）
**目標**: 全ポートフォリオの更新を成功させ、運用を再開する。

1. **ロングラン・負荷テスト**:
    - 銘柄間・ページ間の適切な待機時間（0.5s〜1.5s）を設定し、403エラーが出ないことを確認。
2. **DB整合性チェック**:
    - 更新後のデータが `portfolio_history.db` に正しく永続化されるか確認。

## 3. リスク管理と防衛線

| リスク | 内容 | 対策 |
| :--- | :--- | :--- |
| **解析フリーズ** | 巨大な1行のHTMLによる処理停止。 | BeautifulSoupの前に、正規表現でターゲット周辺のみを切り出す前処理を徹底。 |
| **アクセス遮断** | リクエスト増による403 Forbidden。 | セッションの維持と、成功銘柄のDBキャッシュ優先利用。 |
| **計算精度低下** | 不完全なデータによる誤判定。 | 算出根拠が不足している場合は「判定不能」バッジを出し、誤った数値を算出しない。 |
